# Agent and environment compatibility: audit and refactor plan

## Purpose

This repo has been developed exclusively through Codex (Codex Desktop on macOS), and its agent-facing tooling reflects that: a Codex-specific orchestrator/worker contract is woven directly into `AGENTS.md`, `.codex/`, `scripts/`, and `docs/planning/`. Cursor Cloud Agents and GitHub Copilot's coding agent are now also expected to work against this repo.

This document has two parts:

1. A **fact-finding audit** of every artifact in the repo that is specific to one agent platform or one operator's local machine, versus content that is genuinely agent-agnostic.
2. A **phased refactor plan** to reorganize that content so multiple agent platforms can develop this repo without breaking each other, and so adding a future platform is a bounded, low-risk change instead of another round of ad hoc edits scattered across `AGENTS.md`.

This is explicitly **not** a push for a fully agent-agnostic repo (for example, the Codex orchestrator/worker model is a legitimate, working operational choice and is out of scope to redesign here). The goal is compatibility and non-interference, not uniformity.

## Part 1: Audit

### 1.1 Codex-orchestration-specific artifacts

- `.codex/scripts/orchestrator-worker-worktree.sh` — provisions/cleans up git worktrees under `$HOME/.codex/worktrees/workers/...`, exports `CODEX_ENV_SOURCE_ROOT`/`CODEX_WORKER_*` variables, and implements the `startup-gate`/`review-gate`/`publish-gate` checkpoint validation used by the orchestrator/worker handshake.
- `.codex/environments/environment.toml` — Codex Desktop environment profile (autogenerated), including `CODEX_ENV_SOURCE_ROOT` resolution and Codex UI action declarations.
- `.codex/scripts/check-worker-environment.sh` — worker bootstrap readiness probe, invoked from the Codex environment profile's setup script.
- `AGENTS.md`: the **Orchestrator contract** and **Session workflow** sections in full, and the Codex-specific bullets inside **Environment policy** (validated environment claim, `CODEX_ENV_SOURCE_ROOT`/worktree resolution).
- `docs/planning/agent-orchestration.md`, `docs/planning/worker-dispatch-prompt.md` — pure orchestrator/worker operating procedure and dispatch template (`spawn_agent`, `wait_agent`, `BOOT_CHECKPOINT`/`STARTUP_CHECKPOINT`/`REVIEW_CHECKPOINT`/`PUBLISH_CHECKPOINT`).
- `docs/planning/AGENT_GUIDANCE.md` — mostly the same orchestrator checklist, with a smaller generic-issue-hygiene core mixed in.
- `scripts/agent-check.sh`, `scripts/agent-handoff-check.sh` and their `npm run agent:check` / `npm run agent:handoff` scripts — implementation is platform-agnostic bash (uses `gh`/`jq` against GitHub issue state and `docs/planning/open-issue-order.json`), but the concept (single `agent-ready` queue, orchestrator-driven promotion) is currently documented only as part of the Codex orchestrator contract.
- Vocabulary that only makes sense inside this model: `spawn_agent`, `wait_agent`, `BOOT_CHECKPOINT`, `STARTUP_CHECKPOINT`, `REVIEW_CHECKPOINT`, `PUBLISH_CHECKPOINT`, `orchestrator/live`.

### 1.2 Cursor-specific artifacts

- `.cursor/environment.json` — Cursor Cloud Agent bootstrap (install script + exposed port).
- `AGENTS.md`'s **Cursor Cloud specific instructions** section.
- The `cursor/<slug>-<run-id>` branch-naming convention (platform-assigned, not chosen by the agent).

### 1.3 Copilot-specific artifacts

- `.github/copilot-instructions.md` — required to live at this exact path for GitHub Copilot to pick it up automatically; content is mostly generic product/security rules with a short Codex-tooling footer.
- `.github/ISSUE_TEMPLATE/agent_task.md` — previously said "tasks for Copilot agents" even though it's a generic agent-task template (fixed in this change; see Part 3).
- `.github/workflows/supabase-verify.yml`'s `copilot/**` push-branch trigger.

### 1.4 macOS / Apple-Silicon-specific artifacts (found and partially fixed in this change — see Part 3)

- `scripts/tool-install.sh` previously hard-failed unless `uname` reported `Darwin`/`arm64`, even though Supabase publishes Linux amd64/arm64 releases too.
- `scripts/tool-check.sh` previously defaulted its Supabase `HOME` override to `/private/tmp`, which does not exist as a writable path on Linux (`/tmp` is not a symlink to `/private/tmp` outside macOS).
- Several docs (`AGENTS.md`, `README.md`) asserted the Supabase CLI install path was macOS-Apple-Silicon-only; this was true of the implementation but is no longer accurate after the Part 3 fix.

### 1.5 Generic-but-mislabeled or mixed content

- `AGENTS.md`'s **Start conditions**, **Required preflight**, **Handoff contract**, and **Verification contract** sections mix genuinely agent-agnostic queue/PR/verification norms with Codex-orchestrator-specific phrasing ("dispatching a worker", "worker-owned issue branch").
- `docs/planning/manual-testing-checklist-template.md`, `issues-audit.md`, `issue-update-plan.md`, `recommended-issue-sequence.md`, `target-state-mobile-backend-slice-map.md` are durable planning content with light orchestrator framing or an `agent-ready` reference mixed in.
- `docs/technical/testing-strategy.md` has one Codex-sandbox-specific sentence in an otherwise generic CI policy doc.
- `package.json`'s `name` field (`agents-scaffold-nextjs-moviecal-app`) is scaffold naming rather than product naming — cosmetic, not functional.

### 1.6 Durable, agent-agnostic content (do not touch when addressing platform compatibility)

`src/`, `e2e/`, `test/`, `docs/product/*`, `docs/design/*`, `docs/technical/*` (minus the one line above), the core planning docs (`milestones.md`, `backlog.md`, `decision-log.md`, `implementation-plan.md`, `proposed-github-issues.md`), `supabase/migrations/*`, `supabase/config.toml`, `vercel.json`, `.env.example`, and the standard tool configs (`tsconfig.json`, `playwright.config.ts`, `vitest.config.ts`, `next.config.*`, `eslint.config.cjs`, `postcss.config.cjs`).

### 1.7 Newly discovered interoperability gaps (this session)

These are concrete cases of the general problem the user asked about — one platform's assumptions silently breaking another platform's usage of shared scripts/CI:

1. **`scripts/tool-install.sh` hard-failed on any non-Darwin/arm64 host.** Verified by running it on this session's Linux/x86_64 Cursor Cloud Agent VM before the fix (`exit 1`). Codex's own `AGENTS.md` environment policy already admits "the same tooling is expected to work in Unix-like Codex environments, but that path is not yet validated" — so this wasn't only a Cursor problem, it would have hit a Linux-based Codex worker too.
2. **`scripts/tool-check.sh` defaulted to `/private/tmp`, a macOS-only path.** Verified by reproducing the failure (`mkdir: cannot create directory '/private': Permission denied`) on Linux before the fix.
3. **`.github/workflows/supabase-verify.yml`'s push trigger listed `agent/**` and `copilot/**` but not `cursor/**`.** A push straight to a `cursor/*` branch that touches `supabase/**` would not get a push-triggered CI run (a PR-triggered run would still happen once a PR exists, since the `pull_request` trigger here has no branch filter, but that's a weaker safety net and an easy way for prefixes to silently drift out of sync — exactly what happened here).
4. **`.github/ISSUE_TEMPLATE/agent_task.md` says "for Copilot agents"** even though every other doc in the repo treats this as a generic agent-task template used primarily for Codex today.
5. **Node version mismatch across environments.** CI (`verify.yml`, `supabase-verify.yml`) pins Node 24. The repo had no `.nvmrc`/`.node-version`/`engines` field, so any given agent's Ubuntu/macOS base image supplies whatever Node happens to be preinstalled — this Cursor Cloud Agent VM shipped Node 22.14.0. Nothing failed functionally in this session (typecheck/test/build/e2e all passed on Node 22), but it's an unpinned variable that could silently diverge between local dev, Codex, Cursor, and CI. Now addressed with `.nvmrc` + `package.json engines` (see Part 3); actually aligning every platform's base image to Node 24 is left as a Part 4 follow-up since it may require changing a Cursor-side base image/Dockerfile outside this repo.

None of the above were "one agent breaking another agent's PR" in the sense of overwriting each other's work — the actual risk was **silent capability gaps**: a script or CI trigger that quietly doesn't work (or doesn't run) for whichever platform wasn't the one being used when the script was last touched, discovered only when someone happens to run it on the "wrong" platform.

## Part 2: Target architecture

The core idea: separate a **generic contract** every agent/human reads, from **platform operator guides** that only the relevant platform needs, from **platform config directories** that platform tooling reads mechanically. Right now those three layers are flattened into one 99-line `AGENTS.md`.

```
AGENTS.md                                  # generic contract + a router table into docs/operators/*
docs/operators/
  README.md                                # index; "how do I know which platform I'm on"; template for adding a new platform
  codex.md                                 # orchestrator/worker contract, worktrees, checkpoints, .codex/ tooling (from agent-orchestration.md + AGENT_GUIDANCE.md + Environment policy's Codex bullets)
  codex-worker-dispatch-prompt.md          # moved from docs/planning/worker-dispatch-prompt.md, unchanged content
  cursor-cloud.md                          # moved from AGENTS.md's "Cursor Cloud specific instructions" section
  github-copilot.md                        # new: today this is scattered across copilot-instructions.md's footer + the issue template + the CI branch trigger
  branch-and-ci-conventions.md             # single source of truth: branch prefix -> platform -> which CI workflows must reference it (a stronger version of the table added to AGENTS.md in Part 3)
docs/planning/                             # unchanged except the files that move out above; keeps durable product/queue planning docs
.codex/                                    # unchanged - this is Codex's own mechanically-read config directory
.cursor/                                   # unchanged - this is Cursor's own mechanically-read config directory
.github/copilot-instructions.md           # trimmed to generic product/security rules + a pointer to docs/operators/github-copilot.md
scripts/                                   # unchanged locations; now cross-platform per Part 3
```

Why this shape:

- **`AGENTS.md` stays the single required starting point** for any agent (this matches how Codex, Cursor, and Copilot all discover and prioritize `AGENTS.md`), but it becomes short enough to read in full every session: generic start/queue/verification/security rules, plus a table that says "if you are platform X, also read `docs/operators/X.md` before doing anything else."
- **Each `docs/operators/*.md` file follows the same internal template** (bootstrap/config file location, tool availability quirks, secrets handling, branch convention, known gaps) so adding platform N+1 is "copy the template, fill in what's different," not "read the whole existing `AGENTS.md` and figure out where to wedge in a new section without breaking the others" (which is exactly what happened when the Cursor section was added on top of the Codex-only contract).
- **`branch-and-ci-conventions.md` is the single place** that maps branch prefixes to platforms and to the CI workflows that must know about them, so the `supabase-verify.yml` drift found in Part 1.7 can't happen silently again — a reviewer checking "did we add a new prefix?" only has one file to check against workflow YAML.
- **`.codex/` and `.cursor/` stay exactly where they are.** These aren't candidates for reorganization: each platform's tooling looks for its config at a fixed path, so "agent-agnostic" here just means "don't make one platform's config directory try to also serve another platform."

## Part 3: Phase 0 — safe fixes shipped in this change

Everything below is implemented, tested against a real run in this session, and does not touch the orchestrator/worker contract or require maintainer sign-off on a structural doc reorganization:

- `scripts/tool-install.sh`: detects OS (`Darwin`/`Linux`) and arch (`arm64`/`amd64`) instead of hard-requiring Darwin/arm64; downloads the matching Supabase CLI release; skips the macOS-only `codesign` step on Linux. Verified end-to-end on this session's Linux/x86_64 VM (`npm run tool:install` then `npm run tool:check` both succeed and report real Vercel/Supabase versions).
- `scripts/tool-check.sh`: fixed the `/private/tmp` macOS-only default to a portable `/tmp` default; softened the Supabase-not-runnable error message so it no longer assumes Apple Silicon.
- `.github/workflows/supabase-verify.yml`: added `cursor/**` to the push-branch trigger list, matching the existing `agent/**`/`copilot/**` pattern.
- `.github/ISSUE_TEMPLATE/agent_task.md`: reworded "tasks for Copilot agents" to "any coding agent (Codex, Cursor, GitHub Copilot, or similar)".
- `AGENTS.md`: added the branch-prefix/CI-trigger table described in Part 2 (a lighter-weight version of the eventual `docs/operators/branch-and-ci-conventions.md`), corrected the now-inaccurate "Apple Silicon macOS only" claims in **Environment policy** and the **Cursor Cloud specific instructions** section, and clarified that the `agent/<issue-number>-<short-slug>` branch format is Codex-specific rather than universal.
- `README.md`: corrected the matching "Apple Silicon macOS only" claims about `npm run tool:install`, and added a pointer from "Agentic development setup" to this document and to `AGENTS.md`'s platform-specific sections.
- Added `.nvmrc` (`24`) and `package.json`'s `engines.node` (`>=24`) so any agent platform or local dev setup that respects either convention provisions the same Node major version CI uses.

None of these changes alter the meaning of the orchestrator/worker contract, the `agent-ready` queue invariant, or any branch-naming convention already in use — they only make already-shared scripts, CI triggers, and docs correctly reflect the platforms that actually use them.

## Part 4: Proposed next phases (not yet executed — need a decision, not just an audit)

These are bigger structural moves. They're described here so they can be reviewed and sequenced deliberately, rather than executed inline in this same change, because they touch `AGENTS.md`'s live orchestrator contract while a Codex orchestrator/worker queue may be actively running against this repo.

- **Phase 1 — extract platform sections out of `AGENTS.md` into `docs/operators/*.md`.** Move the Orchestrator contract, Session workflow, and Codex-specific Environment policy bullets into `docs/operators/codex.md`; move the Cursor Cloud section into `docs/operators/cursor-cloud.md`; write `docs/operators/github-copilot.md` from the current `copilot-instructions.md` footer plus the issue-template/CI-trigger facts in Part 1.3. Replace the moved sections in `AGENTS.md` with a short router table. This is a pure move (no semantic change) but touches a file every in-flight agent session reads, so it should land in its own governance PR when no worker session is actively mid-task, per the existing "governance changes stay separate from feature delivery" rule.
- **Phase 2 — consolidate the Codex orchestration docs.** `docs/planning/agent-orchestration.md` and `docs/planning/AGENT_GUIDANCE.md` currently overlap heavily; fold them into the new `docs/operators/codex.md` and either delete the originals or leave short redirect stubs, then update `docs/README.md`'s reading order and every doc that currently links to the old paths (`worker-dispatch-prompt.md`, `AGENT_GUIDANCE.md`, `copilot-instructions.md`).
- **Phase 3 — promote the branch-prefix table into its own `docs/operators/branch-and-ci-conventions.md`** once there is more than one CI workflow that needs to reference it, and add a lightweight CI check (or a step in `agent-check.sh`) that fails if a branch prefix appears in this doc's table but not in a workflow's trigger list, or vice versa, so the Part 1.7 drift class of bug is caught automatically instead of by manual audit.
- **Phase 4 — decide on Node-version alignment across platforms.** `.nvmrc`/`engines` now name Node 24 as the target, but this session's Cursor Cloud Agent VM's base image ships Node 22 by default. Decide whether to align Cursor's environment configuration (e.g., via a Dockerfile that installs Node 24) to the pinned version, or relax the pin to whatever is actually validated everywhere; either is fine, but the two should agree.
- **Phase 5 — decide the queue-eligibility question for non-Codex platforms.** Right now `agent-check.sh`/`agent-handoff-check.sh` and the `agent-ready` queue are Codex-orchestrator-governed. Decide whether Cursor Cloud Agents and Copilot's coding agent should be able to pick up `agent-ready` issues directly (accepting their own branch-prefix conventions), or should be scoped to a different lane (governance/docs/chore work, or ad hoc tasks outside the queue) while Codex's orchestrator/worker system keeps owning the feature-issue queue. This was flagged as an open question in the prior Cursor environment-setup change and is restated here because it's the biggest remaining compatibility decision — everything in Phases 1-3 works either way, but Phase 5 changes what "agent-ready" is allowed to mean.
